# Task 3 - Analytical Data Model

## Nectar Data Engineering Assignment

This document describes the analytical data model designed for the IoT Asset Monitoring System. The data warehouse follows a Star Schema design to support business intelligence, reporting, and future machine learning applications.

## Star Schema Overview

The analytical data warehouse is designed using the Star Schema approach.

The model consists of:

- Dimension Tables
- Fact Tables

Dimension tables store descriptive business information, while fact tables store measurable business events generated by IoT devices.

This design minimizes redundancy, improves query performance, and simplifies reporting.


## Dimension Tables

### dim_site

Stores information about each site.

Columns:
- site_id
- site_name
- city
- industry
- site_type

---

### dim_building

Stores building information for each site.

Columns:
- building_id
- site_id
- building_name

---

### dim_asset

Stores master information about every IoT asset.

Columns:
- asset_id
- building_id
- site_id
- asset_name
- asset_type
- manufacturer
- category
- installation_date
- warranty_expiry

---

### dim_time

Stores calendar information used for analytical reporting.

Columns:
- time_id
- full_date
- hour
- day
- month
- quarter
- year
- weekday


## Fact Tables

### fact_telemetry

Stores IoT telemetry readings collected from assets.

Columns:
- telemetry_id
- time_id
- asset_id
- temperature
- humidity
- pressure
- vibration
- power_consumption
- operating_mode

---

### fact_event

Stores operational events generated by assets.

Columns:
- event_id
- time_id
- asset_id
- event_type
- severity
- message

---

### fact_energy

Stores hourly aggregated energy consumption for each asset.

Columns:
- energy_id
- time_id
- asset_id
- hourly_energy
- average_power

---

### fact_asset_health

Stores the calculated health score and risk level for each asset.

Columns:
- health_id
- time_id
- asset_id
- health_score
- risk_level

---

### fact_maintenance

Stores maintenance recommendations generated from asset health analysis.

Columns:
- maintenance_id
- time_id
- asset_id
- recommendation
- priority
- status


## Partition Strategy

The fact tables are expected to grow continuously as telemetry and event data are generated by IoT assets.

To improve query performance and simplify historical data management, the following partitioning strategy is proposed:

- Partition fact_telemetry by month using the telemetry timestamp.
- Partition fact_event by month using the event timestamp.
- Partition fact_energy by month using the aggregation date.
- Partition fact_asset_health by month using the health assessment date.
- Partition fact_maintenance by month using the recommendation date.

Dimension tables are relatively small and therefore do not require partitioning.


## Indexing Strategy

The following indexing strategy is recommended to improve analytical query performance.

Primary Key Indexes

- site_id
- building_id
- asset_id
- time_id

Additional Indexes

fact_telemetry
- asset_id
- time_id

fact_event
- asset_id
- time_id
- event_type

fact_energy
- asset_id
- time_id

fact_asset_health
- asset_id
- time_id

fact_maintenance
- asset_id
- time_id
- priority

These indexes optimize filtering, joins, and time-based analytical queries.

## Conclusion

The analytical data model follows a Star Schema architecture designed for IoT asset monitoring and predictive maintenance.

The model separates descriptive business entities into dimension tables and operational measurements into fact tables.

In addition to the core analytical model, the design includes asset health monitoring and maintenance recommendation fact tables to support predictive maintenance use cases.

This architecture is scalable, supports business intelligence reporting, and can be extended for future machine learning applications.
